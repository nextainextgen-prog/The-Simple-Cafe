## 3) Task A — Dashboard KPI (หน้าแรกหลังบ้าน)

### ปัญหาตอนนี้

เปิด `/admin` แล้วเจอ dashboard default ของ Payload = การ์ด collection เรียงกันเฉยๆ เจ้าของร้านเปิดมาแล้วไม่รู้ว่า "วันนี้มีอะไรต้องทำ"

### ที่อยากได้

หน้าแรกที่ตอบ 3 คำถามของเจ้าของร้านภายใน 3 วินาที: **มีออเดอร์ใหม่ไหม / ขายได้เท่าไร / มีอะไรค้างต้องจัดการ**

**ไฟล์:** `payload/admin/components/views/Dashboard.tsx` → wire ที่ `admin.components.views.dashboard`

**เป็น Server Component** ดึงข้อมูลด้วย **Payload Local API** (`getPayload({ config })` แล้ว `payload.count()` / `payload.find()`) — ห้ามยิง REST/fetch ตัวเอง ช้าและต้องแนบ auth เอง

### 3.1 แถว KPI (การ์ด 4 ใบ)

| การ์ด | ข้อมูล | query |
|---|---|---|
| ออเดอร์ใหม่ | จำนวน order ที่ยังไม่ได้จัดการ | `count` `orders` where `status = "รอชำระ"` |
| ยอดขายเดือนนี้ | รวมเงิน | sum `total` ของ `orders` ที่ `createdAt` อยู่ในเดือนปัจจุบัน **และ** `status` ∈ `["ชำระแล้ว","กำลังเตรียม","จัดส่ง","สำเร็จ"]` |
| คำขอใบเสนอราคา | lead ที่ยังไม่ติดต่อกลับ | `count` `leads` |
| คะแนนรีวิวเฉลี่ย | ดาว + จำนวนรีวิว | global `social-proof` |

⚠️ **ยอดขายต้องไม่นับ `รอชำระ` และ `ยกเลิก`** — ตัวเลขต้องเป็นเงินที่ได้จริง ไม่ใช่ยอดลม
⚠️ `status` เป็น select ที่เก็บ **ค่าไทยตรงๆ** (`"รอชำระ"`, `"ชำระแล้ว"`, `"กำลังเตรียม"`, `"จัดส่ง"`, `"สำเร็จ"`, `"ยกเลิก"`) ไม่ใช่ slug อังกฤษ — เช็คใน `payload/collections/Orders.ts` ก่อนเขียน query

การ์ด: พื้น surface, ขอบเส้นอุ่น, มุมโค้ง, ไอคอน lucide เส้นโอลีฟมุมหนึ่ง, ตัวเลขเป็นพระเอก (ใหญ่ หนา ink), label เล็กจาง, **กดการ์ดแล้วลิงก์ไปหน้าที่เกี่ยวข้อง** (ออเดอร์ใหม่ → `/admin/collections/orders?where[status][equals]=รอชำระ`)

### 3.2 ออเดอร์ล่าสุด

ตาราง 5-10 รายการล่าสุด: เลขออเดอร์ / ลูกค้า / ยอด / สถานะ (เป็น badge สี) / วันที่ — กดแถวแล้วเข้าหน้า edit ของออเดอร์นั้น

สถานะ badge ใช้สีสื่อความหมายแต่ยังอยู่ในโทน CI: รอชำระ = ทอง/เหลืองอุ่น, สำเร็จ = โอลีฟ, ยกเลิก = เทา/แดงหม่น (อย่าจี๊ด)

### 3.3 ปุ่มลัด

แถวปุ่ม: `+ เพิ่มสินค้า` · `ดูออเดอร์ทั้งหมด` · `แก้เนื้อหาหน้าเว็บ` · `คลังสื่อ` — ปุ่มโค้งมน (`999px` ตามธีม)

### 3.4 เคสที่ต้องไม่พัง

- **ยังไม่มีข้อมูลเลย** (ร้านเพิ่งเปิด/DB ว่าง) → ต้องขึ้น empty state สวยๆ ไม่ใช่ `0` โดดๆ หรือ error
- **DB ล่ม** → dashboard ต้องไม่ทำให้ทั้ง admin ขาว ครอบ try/catch แล้วขึ้นข้อความบอก (ดูแพตเทิร์น static fallback ที่ `lib/cms.ts` ใช้อยู่)
- **editor ที่ไม่ใช่ admin** เปิดมา → ต้องไม่เห็นข้อมูลที่ไม่มีสิทธิ์ ใช้ `req.user` เช็ค (ดูแพตเทิร์น `isAdmin` ใน `payload/collections/Users.ts` — เป็น house style)

### ✅ เกณฑ์ผ่าน Task A

- เปิด `http://localhost:3002/admin` เห็น KPI 4 ใบ + ออเดอร์ล่าสุด + ปุ่มลัด
- **ตัวเลขตรงกับ DB จริง** — พิสูจน์: สร้าง order ทดสอบ 1 รายการ status `ชำระแล้ว` → ยอดขายเดือนนี้ต้องขยับตามยอดนั้นเป๊ะ, สลับเป็น `ยกเลิก` → ยอดต้องลดกลับ
- กดการ์ด/แถว/ปุ่ม ไปหน้าถูกต้องทุกอัน
- ลบข้อมูลออเดอร์หมด → เห็น empty state ไม่ใช่ error
- แนบ screenshot มาใน PR

## 4) Task B — หน้าตั้งค่าแบบการ์ด

### ปัญหาตอนนี้

กลุ่ม `ตั้งค่า` ในเมนูมี global `site-settings` โดดอันเดียว เจ้าของร้านกดเข้าไปเจอฟอร์มยาวเหยียดม้วนเดียวจบ — หาไม่เจอว่าจะแก้เบอร์โทรตรงไหน

### ที่อยากได้

หน้า hub `/admin/settings` เป็น **card grid** แต่ละใบ = หมวดการตั้งค่า มีไอคอน + ชื่อ + คำอธิบายสั้นว่าแก้อะไรได้ กดแล้วเข้าไปแก้

**ไฟล์:** `payload/admin/components/views/SettingsHub.tsx` → wire ที่ `admin.components.views.settings` (route `/admin/settings`)

การ์ดที่ควรมี (อิงของที่มีจริงใน `payload/globals/SiteSettings.ts` — **ไปอ่านก่อนว่ามีฟิลด์อะไรบ้าง** อย่าเดา):

- ข้อมูลร้าน (ชื่อ/โลโก้/ที่อยู่/เวลาทำการ)
- ช่องทางติดต่อ (โทร/LINE/อีเมล)
- เมนูนำทาง + footer
- แถบข้อความวิ่ง (marquee)
- สถิติหน้าแรก
- ผู้ใช้และสิทธิ์ (→ `/admin/collections/users`) — **การ์ดนี้โชว์เฉพาะ admin** (ใช้ `isAdmin` เช็ค)

ถ้า global `site-settings` แยกเป็น tab/group อยู่แล้ว ให้ลิงก์เข้า tab ตรงๆ ได้เลยถ้า Payload รองรับ ถ้าไม่รองรับ ลิงก์เข้าหน้า global เฉยๆ ก็พอ — ไม่คุ้มที่จะรื้อ schema ของ global แค่เพื่อให้ลิงก์สวย แต่ตัดสินใจเองได้ถ้าเห็นว่าคุ้ม

### ✅ เกณฑ์ผ่าน Task B

- `/admin/settings` เป็น card grid ตามธีม ไม่ใช่ลิสต์เปล่า

- login เป็น editor (ไม่ใช่ admin) → ไม่เห็นการ์ด "ผู้ใช้และสิทธิ์"
- แนบ screenshot มาใน PR

## 7) ตัดสินใจเองได้ ไม่ต้องถาม

**ทำให้จบทั้ง 2 งานเลย ไม่ต้องรอคำยืนยันระหว่างทาง** เจอทางแยกให้เลือกเองด้วยวิจารณญาณ แล้ว**เขียนใน PR ว่าเลือกอะไรเพราะอะไร** — รีวิวกันที่ PR ทีเดียว

- API ไม่ตรงกับที่เขียนในไฟล์นี้ → เชื่อ `node_modules/payload/dist/**` (ของจริงในเครื่อง) มากกว่าไฟล์นี้ แล้วบอกใน PR ว่าต่างตรงไหน
- ตัวเลข KPI ไม่ตรง DB → ไล่หาเองจนเจอ **ห้ามส่งงานทั้งที่ตัวเลขยังผิด** ถ้าจนจริงๆ ให้ส่ง PR เป็น draft พร้อมบอกว่าติดตรงไหน
